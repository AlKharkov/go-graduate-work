## Препроцессор обрабатывает программу по следующим правилам:
1. Значения констант подставляются вместо их имён
1. При перечислении констант, переменных, параметров функций типы явно подставляются: (a, b int) -> (a int, b int)
1. Вычисляются и подставляются значения по умолчанию для типизированных значений, объявляемых без значения, а только с типом
1. Вычисляются и подставляются типы по умолчанию для переменных, объявляемых только со значением, но без типа
1. Раскрываются встроенные структуры и интерфейсы
1. В литералах массивов везде подставляются индексы (после индексированного идёт +1 индекс), аналогично для литералов срезов. В литералах структур имена полей подставляются (в порядке определения в типе структуры), если не указаны явно
1. Заменяет все "untyped constant" на "typed value", подставляя тип по контексту. Таким образом любое выражение имеет явный тип.


## Препроцессор конкретика:
1. array lit: вставить индексы, где нужно. Такое же для срезов и структур (в порядке декларации полей в типе структур)
1. Переместить init перед оператором, и заключить всё в новый блок для следующих операторов: if, select, for clause


## Limitations
1. Без шаблонов
1. Поля в одной структуре должны быть уникальными, т.е. если в исходной структуре есть поле, то его имя не совпадает ни с каким именем исходной структуры или любой другой её подструктуры
1. Без каналов, горутин, оператора select - всякой многопоточности
1. float32=float64=LispFloat (conversion)
1. Без интерфейсов (пока что не могу придумать правила для динамических типов) (сделать type switch expr)
1. For range function

1. Значением литералов не может быть только "literal" т.к. не поддерживает указатели
1. Type assertion может вызвать punic
1. Подставляется значение из строки выше только в блоке const()
1. Так ли нужен "parameter name"?
1. Так ли нужен "field name"?
1. Метки хоть и видны во всей функции, по факту могут использоваться только в блоке, где определяются (и вложенных в него). Так что для семантики допустимо говорить, что метка видна только в блоке, так как уникальность и т.п. проверяется компилятором, а у нас входная программа считается корректной
1. При декларации переменных в блоке, запоминается старая ячейка переменной (возможно nil), а после завершения обработки блока все такие декларации возвращаются к прежним значениям. Причём ячейку сохранять нужно непосредственно перед декларацией затеняемой переменной
1. nil имеет атрибут type, но при отсутсвии такого является нетипизированным
1. Каналы работают по принципу очереди FIFO
1. Каналы создаются function call: make()
1. Однонаправленные каналы проверяются компилятором, по семантике не отличаются при корректной программе
1. Константы хоть и остались в модели, но в операционной семантике не используются
1. Любой оператор на вход получает (mo "block") - последний блок, в который должен при декларации сохранять старую ячейку
1. Lvalue: переменные, разыменование, elemetns of array, slice, struct, empty
1. if stmt with init -> block{ init; if stmt without init }
1. switch stmt без init аналогично
1. switch stmt считаю, что default написан последним, или так его помещаю при обработке программы
1. for clause stmt без init аналогично
1. for range map получить список ключей (mot m :amap t1 t2) -> (attributes m)?
