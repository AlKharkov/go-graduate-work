\section{Ограничения фрагмента языка Go}
Полное моделирование языка представляет собой задачу исключительной сложности. Современные языки программирования включают сотни синтаксических конструкций, множество тонких семантических правил и исторические особенности, накопившиеся за годы развития. Разумеется, зачастую достаточно знать лишь те конструкции языка программирования, что используются чаще всего. Ограничение исследуемых конструкций языка Go так же можно обосновать следующими соображениями:
\begin{enumerate}
\item Сокращение общего объёма кода способствует упрощению процесса выявления ошибок и неточностей в программной реализации;
\item Уменьшенный объём и структурированность кода облегчают анализ разработанных решений, позволяют чётко выделить ключевые закономерности, что, в свою очередь, повышает научную ценность проведённого исследования.
\end{enumerate}

Поэтому в данной работе речь пойдёт именно про базовую часть языка Go, без использования каких-либо библиотек или пакетов.

Предварительная обработка исходного кода осуществляется с помощью лексического парсера, преобразующего сырой код в унифицированную форму. Данная операция сопоставима с этапами предварительной обработки, выполняемыми компилятором перед генерацией исполняемого кода. Далее опишем конкретные функции нашего лексического парсера:
\begin{enumerate}
\item Парсер явно указывает типы всех объектов. Это реализуемо потому, что язык Go является строго типизированным, значит на этапе компиляции уже известны типы всех объектов программы;
\item В языке Go, например, при декларации нескольких переменных в одном блоке, допускается опускать значения. В таких случаях парсер самостоятельно подставляет значение с предыдущей строчки блока;
\item Все константные значения вычисляются на этапе компиляции программы. Поэтому парсер убирает все декларации констант, заменяя их имена соответсвующими значениями. В том числе значение iota;
\item Некоторые способы создания пользовательских типов на самом деле являются обычным переименованием уже существующих. Поэтому на этапе компиляции производится подстановка имён типов;
\item Так же вместо функций и интерфейсов с шаблонными типами создаются экземпляры с необходимыми подставленными типами;
\item В языке Go доступна возможность краткого декларирования переменных \colorbox{goBack}{\lstinline[style=Go]{x := 1}}, что с учётом замечаний выше очевидно приводится к полному декларированию переменных \colorbox{goBack}{\lstinline[style=Go]{var x int = 1}};
\item Импорт пакетов в код программы интерпретируется как декларативный механизм указания экспортируемых составляющих пакета - функций, переменных и других элементов - до старта выполнения программы. Доступ к этим элементам осуществляется по составным именам вида <название пакета>.<название элемента>.
\end{enumerate}

\section{Онтология фрагмента языка Go}
Прежде чем разбираться с моделированием действий различных программ или их составляющих, необходиомо предварительно создать модель для элементов, составляющих язык Go, которые присутсвуют в коде программы: переменные, функции, операторы, декларации и т.д. Для создания онтологии фрагмента языка Go, на языке ABML будут созданы типы объектов, соответсвующие некоторым объектам фрагмента языка Go.

Классификация и порядок изложения опираются на официальную спецификацию языка Go. Большая часть названий объектов, моделирующихся в этом разделе, будет совпадать с их названиями в официальной спецификации для удобства восприятия и облегчения поиска дополнительной информации по каждому из них. Однако некоторые сущности были признаны излишними с точки зрения моделирования их функицональности, а некоторые имена недостаточно отражающими суть описываемого объекта. Это вызвало некоторую разницу в именовании нескольких объектов в построенной модели.


\subsection{Имена}
Ниже создаются модели для типа, описывающего имена объектов в языке Go. Отмечу, что для полного понимания изложенного здесь материала необходимо понимание основных команд языка ABML, про который можно подробнее узнать из статей в списке литературы. Однако в данном подразделе будет дано небольшое пояснение по написанному коду на яызке ABML. Ключевое слово \lstinline[style=ABML]{typedef} означает, что данная инструкция моделирует некоторый класс объектов. Следующее за ним слово \lstinline[style=ABML]{"identifier"} указывает на название создаваемого класса. Конструкция \lstinline[style=ABML]{(uniont ...)} показывает объекты, которые могут быть причислены к данному классу. Причём важно отметить, что это определяет необходимые свойства для установления принадлежности некоторого объекта создаваемому класса, но не достаточные. То есть на данный момент при моделировании таких классов зачастую допускается большее число элементов, чем реально будет моделироваться данным классов.

Начнём с введения типов, моделирующих имена различных объектов программы. 
\begin{lstlisting}[style=ABML]
(typedef "identifier" (uniont string))
(typedef "variable name" "identifier")
(typedef "field name" "identifier")
(typedef "function name" "identifier")
(typedef "method name" "identifier")
(typedef "interface name" "identifier")
(typedef "label name" "identifier")
(typedef "type name" "identifier")
(typedef "package name" "identifier")
\end{lstlisting}

Тип \lstinline[style=ABML]{"identifier"} описывает общее понятие имени объекта в программе. Остальные типы показывают отношение имени к более конкретным объектам языка Go. 

Таким образом, \lstinline[style=ABML]{"variable name"} используется для имён переменных, \lstinline[style=ABML]{"field name"} - для имён полей в структурах, тип \lstinline[style=ABML]{"function name"} - для имён функций, тип \lstinline[style=ABML]{"method name"} - для имён методов, тип \lstinline[style=ABML]{"interface name"} - для имён интерфейсов, тип \lstinline[style=ABML]{"label name"} - для имён меток, тип \lstinline[style=ABML]{"type name"} - для имён типов, которые не получилось подставить парсером, тип \lstinline[style=ABML]{"package name"} - для имён пакетов или внешних файлов.


\subsection{Значения и память}
В данном подразделе создаются типы для моделирования типизированных ячеек памяти и значений, которые могут в них храниться. Стоит отметить, что в силу типизированности языка Go, возможно явно указать тип каждого значения.

Начнём с типизированных ячеек памяти и общего понятия возможного значения в языке Go.
\begin{lstlisting}[style=ABML]
(mot "location" :at "type" "type" :at "value" "Go value")
(typedef "Go value" (uniont "literal" "location"))
\end{lstlisting}

*добавить пояснение для type value*
value обозначает значение, хранимое в памяти

Далее моделируются типы для постоянных значений, в том числе комплексных чисел. Такие значения явно вычисляются на этапе компиляции.
\begin{lstlisting}[style=ABML]
(typedef "constant" (uniont "true" "false" int real "complex constant" string symbol))
(mot "complex constant" :at "re" real :at "im" real)
\end{lstlisting}

Дальше речь пойдёт о литералах, которые являются значениями в том числе для составных объектов.
\begin{lstlisting}[style=ABML]
(typedef "literal" (uniont "constant" "composite lit" "function lit" "channel lit"))
(typedef "composite literal" (uniont "array lit" "slice lit" "struct lit" "map lit"))
(mot "array lit" :at "type" "array type" :at "value" (listt "Go value"))
(mot "slice lit" :at "type" "slice type" :at "value" (listt "Go value"))
(mot "struct lit" :at "type" "struct type" :at "value" (cot :amap "field name" "Go value"))
(mot "map lit" :at "type" "map type" :at "value" (cot :amap "identifier" "Go value"))
\end{lstlisting}

Тип \lstinline[style=ABML]{"literal"} определяет понятие литерала в общем.

Тип \lstinline[style=ABML]{"composite literal"} определяет понятие составного литерала.

Типы \lstinline[style=ABML]{"array lit"}, \lstinline[style=ABML]{"slice lit"}, \lstinline[style=ABML]{"struct lit"}, \lstinline[style=ABML]{"map lit"} моделируют объекты, представляющие собой значений для массивов, срезов, структур и карт соотвественно.

В следующем блоке описывается тип функционального литерала и связанные с ним.
\begin{lstlisting}[style=ABML]
(mot "function lit" :at "signature" "signature" :at "body" "block")  ; func(a int) bool { return a < 0 }
(mot "signature" :at "parameters" (listt "parameter decl") :at "variadic parameter" "variadic decl" :at "result" "function result")
(mot "parameter decl" :at "name" "identifier" :at "type" "type")
(mot "variadic decl" :at "name" "identifier" :at "type" "type")  ; func (n ...int){}(1, 2, 3)
(mot "function result" :at "result" (uniont (listt "type") (listt "parameter decl")))
\end{lstlisting}

Тип \lstinline[style=ABML]{"function lit"} моделирует объекты, представляющие собой функции. Причём исходная инструкция в исходном коде не декларирует функцию, а создаёт функциональный объект, без привязки к конкретному имени.

Тип \lstinline[style=ABML]{"signature"} представляет собой сигнатуры функций - список параметров функции и её результат, выходные значения.

Тип \lstinline[style=ABML]{"parameter decl"} описывает каждый параметр в объявлении функции.

Тип \lstinline[style=ABML]{"variadic decl"} используется для получения возможности создавать функции с произвольным количеством аргументов. В исходном коде на языке Go перед типом соответствующего аргуента ставится троеточие, например, \colorbox{goBack}{\lstinline[style=Go]{func(n ...int)}}.

Наконец опишем структуру, позволяющую моделировать значения каналов.
\begin{lstlisting}[style=ABML]
(mot "channel lit" :at "type" "channel type" :at "cap" nat :at "buffer" (listt "Go value"))
\end{lstlisting}

Тип \lstinline[style=ABML]{"channel literal"} позволяет моделировать взаимодейтсвия каналов в программе.


\subsection{Типы данных}
В данном подразделе моделируются типы данных, большая часть которых должна быть интуитивно понятна из названий. Типы можно разделить на базовые - представляющие некоторые числа, символы или строки, и составные типы данных - состоящие зачастую совокупность базовых или моделирующих более сложные концепции хранения данных.

Сперва рассмотрим базовые типы данных.
\begin{lstlisting}[style=ABML]
(typedef "type" (uniont "base type" "composite type"))
(typedef "base type" (uniont "boolean type" "numeric type" "rune type" "string type"))
(typedef "boolean type" (enumt "bool"))
(typedef "numeric type" (uniont "int type" "float type" "complex type"))
(typedef "int type" (uniont "int" "int8" "int16" "int32" "int64" "uint" "uint8" "uint16" " uint32" "uint64" "uintptr" "byte"))
(typedef "float type" (enumt "float32" "float64"))
(typedef "complex type" (enumt "complex64" "complex128"))
(typedef "rune type" (enumt "rune"))
(typedef "string type" (enumt "string"))
\end{lstlisting}

Стоит отметить, что строки выше можно разделить на два вида типов. 

Первые объединяют существующие типы данных общим понятием. К таким типам можно отнести следующие: тип \lstinline[style=ABML]{"type"} моделирует общее понятие типа данных, \lstinline[style=ABML]{"base type"} представляет собой простейшие типы данных (логический, числовой, строковый), \lstinline[style=ABML]{"numeric type"} объединяет числовые типы данных, \lstinline[style=ABML]{"int type"} - целочисленные типы данных. 

Второй вид типов характеризуется явным перечислением входящих в него типов: \lstinline[style=ABML]{"boolean type"} моделирует логический тип данных, \lstinline[style=ABML]{"float type"} - вещественные числа, \lstinline[style=ABML]{"complex type"} - комплексные числа, имеющие возможность описать минумую часть числа, \lstinline[style=ABML]{"rune type"} моделирует тип, элементами которого являются символы, в том числе спецсимволы, например к этому типу относятся \colorbox{goBack}{\lstinline[style=Go]{'a' | '\t' | '\U00101234'}}, и, наконец, последний тип \lstinline[style=ABML]{"string type"} представляет собой тип строк.

Далее перейдём к рассмотрению составных типов данных. Сначала явно перечислим, что понимается под понятием составного типа данных \lstinline[style=ABML]{"composite type"}.
\begin{lstlisting}[style=ABML]
(typedef "composite type" (uniont "array type" "slice type" "struct type" "pointer type" "function type" "interface type" "map type" "channel type"))
\end{lstlisting}
Начнём с простейших из них.
\begin{lstlisting}[style=ABML]
(mot "array type" :at "len" nat :at "element type" "type")
(mot "slice type" :at "element type" "type")  ; var s []int
(mot "struct type" :at "fields" (cot :amap "field name" "type"))
(mot "pointer type" :at "type" "type")
\end{lstlisting}

Тип \lstinline[style=ABML]{"array type"} содержит типы, описывающие списки - структуры данных, которые можно описать как упорядоченный набор элементов одного типа. Стоит явно отметить, что типом элементов массива является именно тип составляющих его элементов. Например, \colorbox{goBack}{\lstinline[style=Go]{var a [2][2]int}} создаёт массив, который может быть представлен в виде квадратной матрицы размера 2, и типом элементов в данном примере является \colorbox{goBack}{\lstinline[style=Go]{[2]int}}  - массив длины 2 из целых чисел.
Тип срезов \lstinline[style=ABML]{"slice type"} содержит типы, описывающие срезы. Срез - структура данных, которую можно представить в виде ссылки на последовательную часть исходного массива.

Тип \lstinline[style=ABML]{"struct type"} содержит типы, позволяет создавать новые объекты, состоит из множества полей. Каждое поле структуры характеризуется именем и типом, аналогично обычной переменной.

Тип \lstinline[style=ABML]{"pointer type"} описывает типы ссылок на исходные объекты.

Далее речь пойдёт о функциях.
\begin{lstlisting}[style=ABML]
(mot "function type" :at "type signature" "type signature")
(mot "type signature" :at "types" (listt "type") :at "variadic type" "type" :at "result" (listt "type"))
\end{lstlisting}
Тип \lstinline[style=ABML]{"function type"} описывает типы функций, который характеризуется своей сигнатурой типов \lstinline[style=ABML]{"type signature"}, которая в свою очередь является совокупностью типов параметров \lstinline[style=ABML]{"types"} и \lstinline[style=ABML]{"variadic type"}, и типом результата функции  \lstinline[style=ABML]{"result"}. Тип \lstinline[style=ABML]{"variadic type"} обозначает неопределённое количество типов, на языке Go такой параметр можно поставить только последним, обозначается троеточием перед типом элементов.

Теперь несколько примеров для лучшего понимания связи синатксиса типа функции в языке Go с моделируемым на языке ABML. Список параметров функции указывается в круглых скобках после ключевого слова \colorbox{goBack}{\lstinline[style=Go]{func}}, результат функции указывается после закрытия круглых скобок, т.е. \colorbox{goBack}{\lstinline[style=Go]{func\("types", "variadic type"\) "result"}}. Например, \colorbox{goBack}{\lstinline[style=Go]{func\(int, int, ...string\) bool}} обозначает тип функций, возвращающих логическое значение, и принимающих первыми двумя парметрами целочисленные значения, а последующими возможно передать произвольное количество строк.

Далее рассмотрим моделирование типа интерфейсов.
\begin{lstlisting}[style=ABML]
(mot "interface type" :at "elements" (cot :amap "method name" "signature"))
\end{lstlisting}

Этот тип представляет собой интерфейсы с разлчиными методами. Ниже приведу пример инициации интерфейса на языке Go.
\begin {lstlisting}[style=Go]
type A interface {
  f(n int)     // "method name" = `f`, "signature" = `(n int)`
}
\end{lstlisting}

Тип интерфейс \lstinline[style=ABML]{"interface type"} представиим как множество, элементы которого можно назвать функциями без указания реализации. Синтаксически интерфейс состоит из названий функций и их сигнатур: параметры функции и возвращаемое значение.

Тип сигнатура \lstinline[style=ABML]{"signature"} можно назвать конкретизированием сигнатуры типов \lstinline[style=ABML]{"type signature"}, так как сигнатура содержит определённые имена параметров.

Осталось определить ещё несколько типов данных.
\begin{lstlisting}[style=ABML]
(mot "map type" :at "key type" "type" :at "element type" "type")
(mot "channel type" :at "direction" "direction" :at "type" "type")
(typedef "direction" (enumt "bidirectional" "send" "receive"))
\end{lstlisting}

Тип словарь (карта) \lstinline[style=ABML]{"map type"} является реализацией хэш-таблицы. Приведём пример такого типа на языке Go: \colorbox{goBack}{\lstinline[style=Go]{map\[string\]int}}. В данном примере ключи таблицы имеют строковый тип, а значениями являются целые числа.

Тип канал \lstinline[style=ABML]{"channel type"} используется преимущественно в многопоточных процессах. Каналы можно представить как переменную с дополнительной особенность - при считывании значения "пустой" ячейки происходит блокирование процесса до тех пор, пока другой процесс не положит в данную ячейку какое-либо значение. Причём каждый канал имеет своё направление \lstinline[style=ABML]{"direction"} - некоторые каналы предназначены только для записи или только для получения данных из него, или всё вместе. Наглядный пример использования каналов на языке Go будет приведён ниже, в конце раздела выражений.

\subsection{Блоки}
Этот подраздел посвящён только одному типу блоков \lstinline[style=ABML]{"block"}, в том числе по причине обособленности с точки зрения описания функциональности.
\begin{lstlisting}[style=ABML]
(mot "block"
    :at "statements" (listt "statement")
    ; semantic attributes
    :at "variable location" (cot :amap "variable name" "location")
    :at "label position" (cot :amap "label" nat)
)
\end{lstlisting}

Атрибут \lstinline[style=ABML]{"statements"} представляет собой список команд из блока, переведённых в модельное представление.

Следующие атрибуты не являются необходимыми для представления типа \lstinline[style=ABML]{"block"}, однако упрощают реализацию действий некоторых команд, включающих в себя взаимодейтсвие с блоками. Поэтому допустимо называть следующие атрибуты семантическими.

Для каждой переменной, инициализируемой в этом блоке, сохраняется её ячейка памяти до начала обработки блока. Если в блоке будет инициализирована переменная с таким же именем, так называемая локальная переменная, то в результате под старым именем будет использоваться новая переменная, а после заверешния обработки блока необходимо будет вернуть старую переменную - соотнести имени переменной её ячейку памяти до начала обработки блока. Если переменной раньше не существовало, значение по умолчанию устанавливается \lstinline[style=ABML]{nil}

Атрибут \lstinline[style=ABML]{"label position"} является отнешением, сопоставляющим каждой метке её позицию в этом блоке.


\subsection{Декларации}
Настало время определиться с декларированием различных объектов. Начнём с общего понятия декларации объекта \lstinline[style=ABML]{"declaration"}.
\begin{lstlisting}[style=ABML]
(typedef "declaration" (uniont "type decl" "var decl" "function decl" "method decl" "interface decl"))
\end{lstlisting}

Далее опишем декларирование типов \lstinline[style=ABML]{"type decl"}
\begin{lstlisting}[style=ABML]
(mot "type decl" :at "name" "type name" :at "type" "type")
\end{lstlisting}

Сразу же приведёт пример соответсвующей конструкции на языке Go: 

\begin{lstlisting}[style=Go]
type Point struct{ x bool }
\end{lstlisting}
В данном примере объявляется тип \colorbox{goBack}{\lstinline[style=Go]{Point}}, элементами которого являются структуры с одним полем \colorbox{goBack}{\lstinline[style=Go]{x}} типа \colorbox{goBack}{\lstinline[style=Go]{bool}}. Причём компилятор не производит подстановку определённых таким образом типов на этапе компиляции программы.

Теперь опишем декларацию переменных, разделяя её на 2 типа: полная декларация \lstinline[style=ABML]{"var decl"} с явным указанием типа переменной, и короткая декларация \lstinline[style=ABML]{"short var decl"}, тип которой устанавливается по соответсвующему начальному значению при декларации.
\begin{lstlisting}[style=ABML]
(mot "var decl" :amap "variable name" "type & value")
(mot "type & value" :at "value" "expression" :at "type" "type")
\end{lstlisting}

Здесь можно привести пример декларации переменной на языке Go: \colorbox{goBack}{\lstinline[style=Go]{var x int = 1}}. Так же стоит отметить, что тип \lstinline[style=ABML]{"type & value"} не является синтаксически уникальным, однако отличается от ранее заданного типа \lstinline[style=ABML]{"location"} с точки зрения семантики языка.

Теперь, перейдём к декларированию функциональных объектов: функций \lstinline[style=ABML]{"function decl"} и методов \lstinline[style=ABML]{"method decl"}.
\begin{lstlisting}[style=ABML]
(mot "function decl" :at "name" "function name" :at "signature" "signature" :at "body" "block")
\end{lstlisting}

Далее приведём пример декларации функции на языке Go:
\begin{lstlisting}[style=Go]
func add(p *Point, q *Point) {
    p.x += q.x
}
\end{lstlisting}

Соответственно декларация методов.
\begin{lstlisting}[style=ABML]
(mot "method decl" :at "receiver" "parameter decl" :at "name" "method name" :at "signature" "signature" :at "body" "block")
\end{lstlisting}

Дальше приведём пример декларации метода на языке Go:
\begin{lstlisting}[style=Go]
func (p *Point) add(q *Point) {
	p.x += q.x
}
\end{lstlisting}

Не трудно заметить, что функции и методы очень похожи между собой.

Наконец, данный подраздел заканчивает декларирование интерфейсов.
\begin{lstlisting}[style=ABML]
(mot "interface decl" :at "name" "interface name" :at "type" "interface type")
\end{lstlisting}


\subsection{Выражения}
Данный раздел посвящён моделированию выражений - некоторых команд, что после исполнения возвращают некоторое значение. Начнём с перечисления типов, являющихся выражениями \lstinline[style=ABML]{"expression"}.
\begin{lstlisting}[style=ABML]
(typedef "expression" (uniont "literal" "conversion" "method expr" "selector expr" "index expr" "slice expr" "type assertion" "function call" "unary expression" "binary expression"))
\end{lstlisting}

Теперь приступим к определению перечисленных выше типов.
\begin{lstlisting}[style=ABML]
(mot "conversion" :at "type" "type" :at "expression" "expression")
(mot "method expression" :at "receiver type" "type" :at "name" "method name")
\end{lstlisting}

Элементы типа \lstinline[style=ABML]{"conversion"} позволяют выполнять приведение типов. К данной конструкции можно привести следующий пример приведения целочисленного значения \colorbox{goBack}{\lstinline[style=Go]{2}} к типу \colorbox{goBack}{\lstinline[style=Go]{float64}}: \colorbox{goBack}{\lstinline[style=Go]{float64(2) == 2.0}}.

Элементы типа \lstinline[style=ABML]{"method expression"} позволяют создавать функции по методам. Такие функции получаются перестановкой получателя метода на место первого параметра функции, таким образом можно говорить об эквивалентности метода и порождаемой им функции.

Далее опишем выражения выбора поля структуры и метода по экзмепляру структуры.
\begin{lstlisting}[style=ABML]
(typedef "selector expression" (uniont "selector struct" "selector method"))
(mot "selector struct" :at "struct" "expression" :at "field" "field name")
(mot "selector method" :at "method" "expression" :at "method" "method name")
\end{lstlisting}

Для выражения выбора поля структуры можем использовать следующий пример на языке Go: \colorbox{goBack}{\lstinline[style=Go]{person.Age}}. Для выражения выбора метода по экземляру структуры можем привести следующее: \colorbox{goBack}{\lstinline[style=Go]{file.Close()}}.

Следующим опишем похожий на 2 предыдущих тип, но в данном случае выбирается элемент индексированной структуры данных.
\begin{lstlisting}[style=ABML]
(mot "index expr" :at "arr" "expression" :at "index" "expression")
\end{lstlisting}

Данное выражение применимо к следующим типам данных: массивы, срезы, карты и строки.

Ранее упоминалось, что срез является ссылкой на исходный массив, теперь введём тип для выражений, позволяющих создавать срезы по уже существующему массиву.
\begin{lstlisting}[style=ABML]
(mot "slice expr" :at "arr" "expression" :at "low" "expression" :at "high" "expression" :at "max" "expression")
\end{lstlisting}

Это выражение содержит не более трёх аргументов, последний из которых указывать не обяательно, посколько он по умолчанию устанавливается равным разности длины исходного массива (или строки - массива символов) и параметра low. Параметр low является индекс в исходном массиве, от которого срез начинается, включительно. Параметр high соответсвенно является конечным индексом, но уже искючительно. Параметр max устанавливает максимально возможную ёмкость среза, таким образом по умолчанию срез может расти вплоть до последнего элемента массива.

Следующая конструкция используется для приведения интерфейсных типов данных, возвращает получилось ли выполнить соответсвующее приведение.
\begin{lstlisting}[style=ABML]
(mot "type assertion" :at "interface" "expression" :at "type" "type")
\end{lstlisting}

В качетсве примера к последней инструкции можно привести следующее:
\begin{lstlisting}[style=Go]
var i interface{} = 3
num := i.(int) // Успех: num = 3 (тип int)
\end{lstlisting}

\begin{lstlisting}[style=ABML]
(mot "function call" :at "function" "expression" :at "arguments" (listt "expression"))  ; min(1, 2)
(mot "<-1" :at 1 "expression")  ; Return the value from the channel 1
\end{lstlisting}

Далее описано создание типов для унарных алгебраических выражений из языка Go.
\begin{lstlisting}[style=ABML]

(mot "*1" :at 1 "expression")
(mot "&1" :at 1 "expression")
\end{lstlisting}

Представленные выше алгебраические выражения имеют привычный, по известным языкам программирования, синтаксис. Так же описание конкретного значения для каждого такого выражения по отдельности не представляет собой смысла для данной работы.
